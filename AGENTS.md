# 屏幕活动记录 Agent 开发说明

## 项目目标

开发一个能在windows操作系统下本地运行的屏幕活动记录 Agent。

Agent 通过定期识别电脑屏幕，判断用户当前正在做什么，并自动生成一天的活动时间线。它不是反拖延工具，不负责提醒、监督、批评或强制干预用户，只负责客观记录和统计。

最终应支持：

- 自动记录每天的活动时间线。
- 活动记录精确到分钟。
- 自动分类事件。
- 合并连续相同活动。
- 支持查询每天、每周、每月的时间占比。
- 查询某一天时，展示详细到各个时间段的活动记录。

记录格式：

```text
时间 —— 事件
```

推荐完整格式：

```text
YYYY-MM-DD HH:mm-HH:mm —— 一级分类 / 二级分类 / 三级分类 —— 事件描述
```

示例：

```text
2026-05-09 13:41-14:35 —— 游戏 / 鸣潮 —— 游玩《鸣潮》
2026-05-09 15:20-16:10 —— 学习 / 编程学习 —— 阅读 Python 教程并写代码
2026-05-09 20:05-20:42 —— 学习 / AI Agent —— 在 B站观看 AI Agent 相关教学视频
2026-05-09 21:10-21:45 —— 娱乐 / 动画 —— 在 B站观看动画
```

## 总体原则

- 只记录事实，不做道德评价。
- 不提醒用户摸鱼。
- 不批评、不讽刺、不劝阻。
- 不关闭软件，不干预用户操作。
- 优先本地存储。
- 对隐私内容做脱敏处理。
- 当识别不确定时，标记为“未知”或“疑似”，不要编造。
- 连续相同活动应合并为一个时间段。
- 查询日报时，必须展示详细时间线。
- 查询周报、月报时，重点展示时间占比和主要活动。
- 数据、数据库在当前位置新建文件夹，保存在这个新建文件夹的子目录下
- 时间以北京时间为准

## 技术路线

- 语言：Python
- 环境管理：uv
- AI 模型：GPT 兼容接口
- API Base URL：`https://apiport.cc.cd`
- API Key：开发阶段可直接从本地 .env读取，以后图形界面可以做一个设置页，输入一次 API Key，然后保存到本地安全存储 .env里
- SDK：OpenAI Python SDK
- 数据存储：第一版使用本地 JSON / JSONL，后续可升级为 SQLite
- 截图方式：调用操作系统原生截图能力
- 默认监控间隔：60 秒
- 记录精度：分钟级

## 阶段一：感知与最小闭环 Perception

### 目标

配置 uv 环境，实现：

```text
截图 -> 转 Base64 -> 发送给 GPT -> 获得屏幕活动分析 -> 终端打印结果
```

### 环境初始化命令

```bash
uv init screen-activity-agent
cd screen-activity-agent
uv add openai pillow
```

### API 调用要求

使用 OpenAI Python SDK 的 GPT 兼容接口。

客户端初始化示例：

```python
import os
from openai import OpenAI

client = OpenAI(
    api_key=os.getenv("OPENAI_API_KEY"),
    base_url="https://api.openai.com/v1",
)
```

### 第一阶段功能要求

编写 Python 代码实现：

- 调用系统原生截图功能。
- 将截图转为 PNG。
- 将 PNG 转为 Base64。
- 发送给 GPT 视觉模型。
- 要求模型识别当前屏幕内容。
- 输出当前活动分类和事件描述。
- 在终端打印识别结果。

### 第一阶段 Prompt 要求

Prompt 应要求 AI 返回当前屏幕活动分析，内容包括：

- 当前可能使用的软件。
- 当前屏幕主要内容。
- 用户当前可能正在做的事情。
- 一级分类。
- 二级分类。
- 三级分类，可为空。
- 简短事件描述。
- 置信度。

注意：不要把“视频”作为一级分类。视频只是内容载体，AI 应根据视频内容和用户目的判断分类。例如，观看编程教程应分类为“学习”，观看技术会议回放可分类为“工作”或“学习”，观看动画、综艺、游戏实况等应分类为“娱乐”。

不要判断用户是否摸鱼。

不要评价用户行为。

## 阶段二：短期记忆系统 Memory

### 目标

让 Agent 具备时间维度，能够根据最近几次记录判断当前活动是否延续，并为后续事件合并做准备。

### 持续监控

程序启动后，每隔 60 秒识别一次屏幕。

默认间隔可以通过配置修改。

### 记忆队列

创建一个全局滑动窗口，只保留最近 5 次识别记录。

每条记录包含：

```json
{
  "timestamp": "2026-05-09T13:41:00+08:00",
  "app": "Wuthering Waves",
  "category": ["游戏", "鸣潮"],
  "summary": "用户正在游玩《鸣潮》",
  "confidence": 0.92
}
```

### Prompt 升级

将最近 5 次记录转换为文本，作为上下文提供给 AI。

让 AI 结合：

- 当前截图。
- 当前时间。
- 最近几次活动记录。

判断当前活动是否为上一活动的延续。

### 第二阶段输出要求

AI 应返回结构化结果：

```json
{
  "app": "当前应用或未知",
  "screen_content": "屏幕主要内容摘要",
  "category": ["一级分类", "二级分类", "三级分类"],
  "event": "事件描述",
  "is_continuation": true,
  "confidence": 0.0
}
```

## 阶段三：结构化记录与事件合并 Logging

### 目标

将识别结果从“临时观察”升级为“可查询的活动记录”。

### 强制结构化输出

要求 AI 必须返回纯 JSON，不要返回 Markdown，不要返回解释文字。

JSON 格式：

```json
{
  "timestamp": "2026-05-09T13:41:00+08:00",
  "app": "Wuthering Waves",
  "window_title": "鸣潮",
  "screen_content": "游戏主界面，角色正在移动",
  "category": ["游戏", "鸣潮"],
  "event": "游玩《鸣潮》",
  "is_continuation": true,
  "confidence": 0.92,
  "privacy_risk": false
}
```

### 事件合并规则

如果新识别结果与上一条事件满足以下条件，则合并：

- 一级分类相同。
- 二级分类相同。
- 事件语义相近。
- 两次记录间隔不超过设定阈值。
- AI 判断 `is_continuation = true`。

合并后的事件格式：

```json
{
  "date": "2026-05-09",
  "start": "13:41",
  "end": "14:35",
  "category": ["游戏", "鸣潮"],
  "event": "游玩《鸣潮》",
  "duration_minutes": 54,
  "confidence": 0.91
}
```

### 存储要求

保存两类数据：

1. 原始识别记录。
2. 合并后的活动事件。

推荐目录：

```text
data/
  raw/
    2026-05-09.jsonl
  events/
    2026-05-09.json
```

## 阶段四：分类体系与隐私保护 Classification & Privacy

### 目标

建立稳定的活动分类体系，并避免记录敏感内容。

### 一级分类

默认一级分类包括：

- 工作
- 学习
- 编程
- 写作
- 阅读
- 沟通
- 会议
- 娱乐
- 游戏
- 音乐
- 购物
- 生活事务
- 系统操作
- 空闲
- 未知

### 分类示例

```text
游戏 / 鸣潮
游戏 / Steam / 艾尔登法环
学习 / B站 / 编程教程
学习 / AI Agent / 教学视频
娱乐 / B站 / 动画
娱乐 / B站 / 综艺
娱乐 / 直播 / 游戏实况
学习 / 英语 / 单词背诵
编程 / 前端开发 / React
工作 / 文档 / 周报
沟通 / 微信
沟通 / Discord
购物 / 淘宝
系统操作 / 文件管理
```

### 视频内容分类规则

“视频”不作为一级分类。识别到用户正在看视频时，应根据内容目的归入已有一级分类：

- 看编程、语言、课程、论文讲解、考试资料等，归为“学习”。
- 看工作会议回放、业务培训、产品演示等，归为“工作”或“会议”。
- 看动画、电影、综艺、娱乐直播、游戏实况等，归为“娱乐”。
- 看音乐 MV、演唱会、纯音乐播放等，归为“音乐”。
- 无法判断视频内容时，优先使用“未知”，不要仅因为它是视频就归类为娱乐。

### 隐私规则

Agent 不应记录以下内容原文：

- 密码。
- 验证码。
- 私密聊天全文。
- 银行卡号。
- 身份证号。
- 家庭住址。
- 医疗隐私。
- API Key。
- Token。
- Cookie。

当屏幕包含敏感内容时，应记录为：

```text
隐私内容 / 敏感页面 —— 用户正在查看包含敏感信息的页面
```

不要保存敏感文字原文。

### Prompt 要求

要求 AI：

- 识别屏幕内容，但不要摘录敏感文本。
- 对聊天、邮件、文档只概括活动，不保存正文。
- 如果无法确定内容，分类为“未知”。
- 不要根据少量信息过度推断。

## 阶段五：查询与统计 Reporting

### 目标

支持用户查询每天、每周、每月的活动情况。

### 查询某一天

当用户查询某一天时，返回：

- 当天完整时间线。
- 每个时间段的活动。
- 一级分类时间占比。
- 二级分类时间占比。
- 未知时间占比。

示例：

```text
2026-05-09 活动记录

09:12-09:47 —— 学习 / 编程学习 —— 观看 Python 教程并练习代码
10:03-10:26 —— 沟通 / 微信聊天 —— 在微信中与他人交流
13:41-14:35 —— 游戏 / 鸣潮 —— 游玩《鸣潮》
15:20-16:10 —— 工作 / 文档写作 —— 编辑项目方案文档

时间占比：

工作：50 分钟，23.8%
学习：35 分钟，16.7%
沟通：23 分钟，11.0%
游戏：54 分钟，25.7%
未知：48 分钟，22.8%
```

### 查询某一周

当用户查询某一周时，返回：

- 周总记录时长。
- 一级分类占比。
- 二级分类 Top 活动。
- 每日简要趋势。

示例：

```text

总记录时长：48 小时 20 分钟

2. 工作 / 文档写作：6 小时 40 分钟
```

### 查询某一月

当用户查询某一月时，返回：

- 月度总览。
- 一级分类占比。
- 二级分类 Top 活动。
- 每周趋势。
- 未知时间比例。

示例：

```text

1. 工作 / 文档写作：21 小时 40 分钟
2. 游戏 / 鸣潮：18 小时 25 分钟

```

## 阶段六：图形化界面 GUI

### 目标

项目需要提供图形化界面，不能只依赖命令行。

GUI 的目标是让用户可以直观看到：

- Agent 当前是否正在运行。
- 今日活动时间线。
- 今日分类时间占比。
- 每周 / 每月等统计。
- 原始记录与合并事件。
- 配置项。
- API Key 环境变量是否已配置。

### GUI 技术选择

第一版优先使用 Python 桌面 GUI。

默认推荐使用 `PySide6`，因为它适合做完整桌面应用，界面扩展性好。

安装命令：

```bash
uv add pyside6
```

如果需要绘制统计图表，可以后续增加：

```bash
uv add matplotlib
```

### 首页 Dashboard

首页展示：

- 当前运行状态：运行中 / 已暂停 。
- 最近一次识别时间。
- 最近一次识别结果。
- 今日已记录时长。
- 今日主要分类占比。
- 开始 / 暂停 按钮。
- 手动识别一次按钮。

示例：

```text
状态：运行中
最近识别：2026-05-09 20:31
当前活动：学习 / AI Agent / 编程视频
今日已记录：6 小时 25 分钟
```

### 今日时间线 Timeline

展示当天完整记录。

示例：

```text
09:12-09:47 —— 学习 / 编程学习 —— 观看 Python 教程并练习代码
10:03-10:26 —— 沟通 / 微信聊天 —— 在微信中与他人交流
13:41-14:35 —— 游戏 / 鸣潮 —— 游玩《鸣潮》
20:05-20:42 —— 学习 / B站 / AI Agent —— 观看 AI Agent 教学视频
21:10-21:45 —— 娱乐 / B站 / 动画 —— 观看动画
```

要求：

- 支持按日期切换。
- 支持刷新。
- 支持查看某条记录详情。
- 支持手动修正分类和事件描述。
- 支持手动合并或拆分事件。

### 统计页面 Statistics

支持：

- 日统计。
- 周统计。
- 月统计。
- 近7日统计。
- 近30日统计。

展示内容：

- 总记录时长。
- 一级分类时间占比。
- 二级分类 Top 活动。
- 未知时间比例。
- 每日趋势。

图表建议：

- 饼图：分类占比。
- 条形图：Top 活动。
- 折线图：每日趋势。

第一版可以先用表格和文本展示，后续再增加图表。

### 配置页面 Settings

配置项包括：

- API Base URL。
- 模型名称。
- 截图间隔。
- 数据保存目录。
- 是否保存原始截图。
- 隐私保护开关。
- 敏感内容过滤开关。
- 是否开机自启。

API Key 不应全部明文显示在界面中，可以显示前后的几位例如`sk-cd1...727f` ，然后提供一个复制的按钮复制整个API Key。

```text
OPENAI_API_KEY：已配置 / 未配置
```

如果未配置，应提示用户去系统环境变量中配置。

### 记录管理页面 Records

支持：

- 查看原始识别记录。
- 查看合并后的活动事件。
- 按日期筛选。
- 按分类筛选。
- 删除某条错误记录。
- 手动合并 / 拆分事件。
- 导出 CSV / JSON。

### GUI 行为要求

- 启动 GUI 后，用户可以点击“开始记录”。
- 点击“暂停”后，停止截图和 API 调用，但保留已有数据。
- GUI 不应卡死，截图和 API 请求必须在后台线程执行。
- 如果 API Key 未配置，禁止开始记录，并在设置页提示。
- 如果 API 请求失败，应在界面显示错误，而不是让程序崩溃。
- GUI 关闭时，应询问用户是退出程序，还是继续后台运行。

### GUI 与后台 Agent 的关系

推荐架构：

```text
GUI 主线程
  ├── Dashboard / Timeline / Statistics / Settings / Records
  └── Agent Worker 后台线程
        ├── 定时截图
        ├── 调用 GPT
        ├── 写入 raw 记录
        ├── 合并 events
        └── 通知 GUI 刷新
```

要求：

- GUI 只负责展示和控制。
- Agent Worker 负责截图、识别、存储和合并。
- 数据层独立，方便以后改成托盘应用、后台服务或 Web UI。
- GUI 主线程不得直接执行耗时 API 请求。

## 阶段十：后台运行 Deployment（以linux为例，但实现在windows）

### 目标

让 Agent 可以在关闭终端后长期运行，并为 GUI 的“继续后台运行”能力做准备。

### start.sh

功能：

- 检查进程是否已存在。
- 后台启动主程序。
- 日志输出到 `agent.log`。
- 保存 PID 到 `agent.pid`。

### stop.sh

功能：

- 读取 `agent.pid`。
- 停止后台运行的 Agent。
- 删除 PID 文件。
- 避免误杀无关 Python 进程。

### 使用方式

```bash
chmod +x start.sh stop.sh
./start.sh
./stop.sh
```

## Agent 输出风格

Agent 查询结果应客观、简洁、可统计。

禁止输出：

- “你今天太摸鱼了”
- “你应该更努力”
- “建议你减少游戏时间”
- “你浪费了很多时间”

允许输出：

- “今天游戏类活动共 54 分钟，占已记录时间的 25.7%。”
- “今天 13:41-14:35 主要活动是游玩《鸣潮》。”
- “本周占比最高的活动是工作 / 文档写作。”

## 默认行为总结

Agent 正常运行时应：

1. 定期截图。
2. 调用视觉模型识别屏幕活动。
3. 生成结构化记录。
4. 合并连续相同活动。
5. 保存本地日志。
6. 通过 GUI 展示运行状态、时间线和统计信息。
7. 在用户查询时生成日报、周报或月报。

Agent 不应：

- 主动提醒用户。
- 主动评价用户。
- 主动干预用户。
- 强制关闭软件。
- 记录敏感内容原文。

## 更新日志维护要求

项目必须维护 `CHANGELOG.md`。

每次开发中只要发生以下任一情况，都必须在同一次变更中更新 `CHANGELOG.md`：

- 添加功能。
- 修改功能行为。
- 删除功能。
- 调整配置、运行方式、打包方式或数据格式。
- 修复会影响用户使用体验的问题。

更新日志应包含：

- 日期。
- 变更类型：添加 / 修改 / 删除 / 修复。
- 简洁描述：说明这次变更对用户或系统行为造成了什么影响。

禁止只改代码而不记录功能变化。
